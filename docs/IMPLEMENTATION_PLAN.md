# План доведения до ТЗ заказчика

Дата составления: 2026-06-14. Источник требований: `Запрос.docx` + эталоны из папки
«Ассессмент центр 1» (образец отчёта `.pptx`/`.pdf`, шаблон ИПР `.pdf`, шкала `.docx`,
4 упражнения с блокнотами и инструкциями).

Цель: довести бота до состояния «как в ТЗ и как в образце заказчика» — точное
воспроизведение отчёта и ИПР, таймкоды в блокноте, работа на полном центре из нескольких
упражнений.

Стек не меняем: всё на AI Tunnel + gpt-4o-mini.

---

## Где мы сейчас (база)

Реализовано и совпадает с ТЗ: пайплайн аудио→роли→блокнот→отчёт→ИПР→PDF; структура
блокнота A–F; логика `+/−/НЗ`; формула уровня и шкала; изоляция реплик участника; сквозная
агрегация сильные/зоны роста; LLM-рекомендации; чистка текстов; параллельные LLM-вызовы.

Главные расхождения с образцом: отчёт устроен беднее (наш DOCX vs их 23-слайдовая
презентация с матрицей, бенчмарком, курированной литературой/курсами/заданиями по каждой
компетенции и «отчётными» формулировками); нет таймкодов в блокноте; мультиупражнения не
протестированы; ИПР не воспроизводит их шаблон.

---

## Решения (зафиксированы 2026-06-14)

1. **Формат отчёта:** на выбор пользователя — у бота кнопка «в каком формате выдать:
   **PPTX / DOCX / PDF**». Генерируем данные отчёта один раз, рендерим в выбранный формат.
2. **Рекомендации и задания:** **курированная библиотека заказчика + LLM-персонализация**
   под конкретные зоны роста участника.
3. **Формулировки сильных сторон / зон роста:** **через LLM** — переписывать индикаторы
   блокнота в «отчётный» стиль образца («может упускать из внимания…»).
4. **Тесты способностей:** **убираем из генерации** (нет источника данных — пайплайн только
   аудио). Блок делаем опциональным/выключенным; на сводном слайде прочерк или блок не
   выводится. Можно включить позже, если появятся данные тестов.

## Phase 1 — Данные и точность (критерии успеха ТЗ)

- [x] 1.1 Таймкоды из STT: AI Tunnel Whisper переведён в `verbose_json`, сегменты со временем
      сохраняются на записи (`InterviewRecord.transcript_segments`, миграция

      `/process_exercise` и `/fill_notebook`. Role labeling не трогали.
- [ ] 1.3 Проверка критерия на боевом прогоне: каждый «+» получает `[мм:сс] «цитата»`.
- [ ] 1.4 Мультиупражнения: прогнать несколько упражнений одного участника, проверить
      сквозную агрегацию (сильные = «+» во всех, где замерялись; зоны роста = любой «−» или
      «+ в одном / − в другом»), формула без НЗ.
- [ ] 1.5 Матрица «компетенция × упражнение»: из блокнотов определить, какие компетенции в

## Phase 4 — ИПР по шаблону заказчика

Структура шаблона (из аудита 0.4):

1. **Методический раздел** (статичный текст, воспроизводим как есть): «Что такое ИПР»,
   «Почему важно составлять ИПР» (6 пунктов), «Пошаговое руководство» (5 шагов: чёткие цели,
   самооценка, план действий, согласование, реализация и оценка), модель 70/20/10.
2. **Заполняемый шаблон** (таблицы):
   - Шапка: ФИО сотрудника, Должность сотрудника, ФИО руководителя, Должность руководителя,
     Период выполнения ИПР, Дата заполнения.
   - На каждую «Область развития №N» (= компетенция-зона роста):
     - `Компетенция:` (название), `Цель развития:` (LLM по зонам роста);
     - таблица с колонками **Действия | Ресурсы | Сроки | Ожидаемые результаты | Оценка о
       выполнении**; строки «Действия» — 5 фиксированных категорий:
       1. Развитие на рабочем месте
       2. Специальные задачи и проекты
       3. Поиск обратной связи
       4. Наставничество
       5. Самообучение

Задачи:

- [ ] 4.1 Шаблон в DOCX (таблицы) через `python-docx`; PDF/PPTX по той же кнопке формата.
- [ ] 4.2 Наполнение: на компетенцию-зону роста — «Цель развития» (LLM) + раскладка действий
      по 5 категориям (LLM 70/20/10), «Ресурсы» = курсы/литература из `competency_library.json`
      (в «Самообучение»). «Сроки» и «Оценка о выполнении» оставляем пустыми (заполняются при
      исполнении), «Ожидаемые результаты» — LLM.
- [ ] 4.3 Экспорт в выбранный формат.
