## 2026-06-22 (поздний вечер) — Чанкование по длительности (HTTP 524)

Тест на упражнении «Планирование»: аудио 45 минут → расшифровка упала с `AI Tunnel Whisper
returned HTTP 524`. 524 — таймаут шлюза (Cloudflare у AI Tunnel). Причина: наше чанкование
считало длину куска **только по размеру** (25 МБ при 32k ≈ 88 минут на кусок), поэтому 45-минутная
запись влезала в один кусок и уходила одним запросом, который Whisper не успевал отдать за таймаут.

Добавил **ограничение по длительности**: `WHISPER_CHUNK_MAX_SECONDS=600` (10 мин). Теперь
`prepare_audio_chunks_for_upload` пробит длительность и, если запись длиннее лимита, режет по времени
**даже если она влезает в 25 МБ**. `split_audio_into_chunks` дополнительно кап `chunk_seconds` по
этому лимиту. 45 мин → 5 кусков по ~10 мин, каждый расшифровывается за ~40с (хорошо в пределах
таймаута), потом склейка с дедупом по нахлёсту (15с). Если ffprobe недоступен — фолбэк на старую
логику по размеру.

## 2026-06-22 (вечер) — Проверка цитат после первого полного теста

Первый сквозной тест на «Беседе с сотрудником» прошёл: уровни считаются верно (проверил
формулу вручную), жёлтые ячейки заполнены и по компетенции, и по подгруппам, отчёт и ИПР —
по образцу. Но нашёл проблему: часть «+» подтверждена **выдуманными цитатами** — модель брала
пример из самого описания индикатора (текст в скобках) и выдавала за речь участника. Признак —
у таких цитат нет таймкода (матчер не нашёл их в записи).

Чиним детерминированно: `verify_evidence_quotes()` сверяет каждую цитату «+» с потоком слов
транскрипта (самая длинная дословная подцепочка). Не нашлась — цитата выбрасывается; если у «+»
не осталось ни одной подтверждённой цитаты → понижаем до «-» (без комментария), чтобы не врать в
уровнях. Порог: длинная цитата (>5 слов) должна иметь ≥4 подряд идущих дословных слова, короткая —
полное совпадение. Вызывается в обоих путях (`workflow`, legacy `notebook`) перед таймкодами и
заполнением. Плюс ужесточил промпт анализа: цитата только дословная, примеры из скобок описания
индикатора брать запрещено. Заодно починил слипшийся заголовок отчёта (`add_break` вместо `\n`).

Роль-своп на первом приветствии («вызывали?» ушло в участника) остаётся, но он минорный и его
ложный «+» подчищается проверкой цитат.

Ещё из этого теста: отправка готового отчёта упала с `TelegramNetworkError: Request timeout`
(локальный Bot API подвис на отдаче файла). Сам отчёт сгенерился — упала только отправка.
Скачивания уже были с ретраями, а отправки документов — нет. Добавил `send_document_with_retry()`
в `telegram_files` и перевёл на неё все 6 мест (блокнот/отчёт/ИПР/расшифровка). Таймаут на попытку —
отдельный `TELEGRAM_FILE_SEND_TIMEOUT_SECONDS=180` (файлы мелкие, незачем ждать как при скачивании
60 МБ): подвисшая попытка падает быстро, ретрай открывает новое соединение. Глобальный обработчик
с кнопками остаётся как последний рубеж, если все попытки не прошли.

## 2026-06-22 — Контекст упражнений из инструкций заказчика

### Что делали

Заказчик дал инструкции к упражнениям (инструкция ведущего, инструкция наблюдателя/ролевого
игрока, инструкции участника). Раньше система их не использовала — видела только аудио + блокнот,
поэтому статус «НЗ» («индикатор не мог наблюдаться») модель фактически угадывала, а роли путались.

Разобрал папки центра (`D:\load\Ассессмент центр 1`): реальные упражнения с инструкциями —
**Беседа с сотрудником**, **Планирование** («Планы руководителя»), **Саморазвитие** (интервью).
Папка «Аналитический кейс» по факту содержит дубль файлов Планирования (добавил как алиас).

**Сначала** сделал зашитую библиотеку: из docx-инструкций вытащил по каждому упражнению сценарий,
поведение ролевого игрока, роль участника, проявления для «НЗ» и компетенции → сложил в
`bot/resources/exercise_library.json` (по образцу `competency_library.json`).

**Потом** владелец напомнил: по ТЗ (раздел 2 «Входные данные») инструкции участников, наблюдателей и
ведущего — это **файлы, которые загружаются в бота** («Файл pdf»). Поэтому переделал на **загрузку**:
теперь в гайде после ввода названия упражнения бот ждёт инструкции (PDF/DOCX, можно несколько файлов),
читает текст и кладёт его в упражнение. Зашитую библиотеку оставил как **фолбэк** — если файлы не
загрузили, для известных упражнений контекст всё равно подставится (UX проще, ничего не ломается).

Почему инструкции стоят **до аудио**: разметка ролей запускается сразу после расшифровки (в воркере
`media_jobs`), поэтому инструкции должны быть в базе ещё до загрузки аудио — иначе роли уже размечены.

Что добавил:
- `bot/services/instruction_files.py` — извлечение текста из PDF (`pypdf`) и DOCX (`python-docx`),
  `append_instructions` (склейка нескольких файлов с тегом-именем, кап 16k символов).
- Колонка `exercises.instructions_text` + миграция `20260622_0011`.
- Гайд: новый стейт `awaiting_instructions`, хендлеры на файл/текст, кнопки «Дальше»/«Пропустить».
- `bot/services/exercise_context.py` — `build_exercise_analysis_block` и `build_role_labeling_hint`
  теперь принимают `instructions_text`: если он есть — берём загруженные инструкции, иначе библиотеку.
- Проводка `instructions_text`: `media_jobs._exercise_context` (3-tuple) → разметка ролей;
  `workflow.run_exercise_processing` и legacy `notebook.py` → анализ блокнота.

В system-prompt анализа добавлено правило: если упражнение по сценарию не создаёт ситуацию для
индикатора — ставь «НЗ», а не «-». В промпт ролей подмешивается, кто ролевой игрок («ведущий»), а кто
оцениваемый («участник»).

### Проблемы и как решали

- Локального Python на Windows нет — синтаксис проверял глазами, JSON провалидировал через
  `ConvertFrom-Json`. Полная проверка — на сервере.
- Циклов импорта нет: `exercise_context`/`instruction_files` не тянут ничего из `bot`.
- Токен-бюджет: инструкции идут в каждый батч анализа и каждый чанк ролей, поэтому кап 16k символов.

### Результат

Инструкции теперь реально загружаются в бота файлами (как в ТЗ), и их текст доходит до разметки ролей
и анализа блокнота. Если файлы не загрузили — работает зашитая библиотека (известные упражнения) или
вообще без контекста (как раньше).

### Заметки на будущее

- Старый `.doc` (бинарный, в папке «Аналитический кейс») не читается — только `.docx`/`.pdf`/`.txt`.
- На сервере поставить зависимость `pypdf` (`pip install -r requirements.txt`) и накатить миграцию.
- Проверить: «НЗ» осмысленнее, роли стабильнее; что несколько файлов инструкций склеиваются.

## 2026-06-14 … 2026-06-16 — Обкатка на тестах, упрощение UX (гайд), доводка по блокноту

После сборки всех фаз (предыдущая запись) пошёл цикл «тест → правка → тест» на реальных
материалах заказчика, плюс по просьбе владельца упростили работу с ботом до кнопок.

### Что делали

**Доводка качества по результатам тестов:**
- **Роли.** Перевели разметку ролей на `gpt-5-mini` (gpt-4o-mini путал 3 голоса без диаризации).
  Затем усилили промпт: жёсткое правило резать длинные склейки Whisper, явное чередование
  «вопрос ассессора → ответ участника» в блоке разбора. Сегментация стала заметно чище,
  число «+» выросло → уровни стали выше и точнее. Остались краевые мислейблы на стыках
  «ответ+вопрос в одном сегменте Whisper» — это уже предел без диаризации (отложили).
- **Таймкоды** подтверждены на боевом прогоне (31 шт `[мм:сс] «цитата»` в Excel).
- **PDF** сделали устойчивым: явные фильтры impress/writer, парсинг пути из вывода LibreOffice,
  glob-fallback и повторная попытка (первый запуск impress с чистым профилем мог вернуть успех
  без файла).
- **Команда моделей:** разные модели под задачи — `gpt-5-mini` (роли), `gpt-4o-mini` (анализ
  блокнота, рекомендации).

**Упрощение UX (по просьбе владельца):**
- Убрали **выбор STT** — аудио всегда идёт на AI Tunnel Whisper (без кнопок Yandex/NeuroAPI).
- Сделали **кнопочный гайд (FSM)**: `/start` показывает меню (расшифровать / оценить одно
  упражнение / оценить все упражнения / помощь), бот ведёт по шагам и подсказывает, что отправить.
  Чтобы пережить асинхронную расшифровку, media-задача получила `exercise_id` (миграция 0010):
  запись авто-привязывается к упражнению, роли размечаются с именем участника, затем бот просит
  блокнот. После обработки — кнопки «➕ Ещё упражнение / 📄 Сформировать отчёт». Команды
  оставили параллельно. Логику отчёта/ИПР/обработки вынесли в переиспользуемые функции.
- Удалили **legacy `/assess`** (оценивал по 8 дефолтным компетенциям, не связанным с моделью
  заказчика — сбивал). Поправили слипшийся заголовок в DOCX-отчёте.

**Находки на тестах и доводка под блокнот заказчика:**
- Блокнот «Беседа с сотрудником» имеет **нестандартную раскладку колонок** (Проявления=C,
  Комментарии=D, Уровень=E, а не типовые D/E/F) — ридер корректно автоопределяет.
- **Жёлтые ячейки уровней на ДВУХ уровнях**: компетенция (5) И подгруппы поведенческих
  проявлений (12). Заполняли только компетенцию — добавили расчёт и запись уровня **по каждой
  подгруппе** (та же формула, НЗ исключается).

### Проблемы и как решали

- **Кнопки «дальше» не появлялись** после обработки упражнения в гайде — блок после обработки
  обращался к данным FSM и падал до показа кнопок. Сделали: всегда показывать кнопки, без
  хрупких обращений.
- **Обрывы скачивания файлов** из локального Telegram Bot API (`ServerDisconnectedError`/timeout):
  контейнер `telegram-bot-api` 8+ дней `unhealthy` (ложный healthcheck в `--local`), дропает
  соединения к медиа-серверам Telegram. Загрузка блокнота была без ретраев → падала с первого
  раза. Перевели её на тот же retry-загрузчик, что и аудио. Слои автономности (рестарт-политика,
  ночной cron, autoheal после фикса healthcheck) обсудили, отложили на потом.

### Результат

- Полный сценарий упражнения проходит через **кнопочный гайд** end-to-end; качество боевое.
- Блокнот заполняется полностью по структуре заказчика (статусы, цитаты с таймкодами, уровни
  компетенций И подгрупп).
- Роли стабильно корректны, оценка осмысленная.

### Заметки на будущее

- Диаризация (WhisperX/провайдер со speaker labels) — закрыть последние мислейблы на стыках.
- Инфра-автономность Telegram Bot API: починить healthcheck + autoheal или ночной рестарт.
- Прогон полного центра из нескольких упражнений (синтетический участник) — проверить матрицу
  и сквозной отчёт.
- Косметика PPTX под брендбук заказчика.

## 2026-06-14 — Доведение под ТЗ заказчика: таймкоды, библиотека, презентация, ИПР

Получили материалы заказчика (ТЗ + эталоны: образец отчёта на 23 слайда, шаблон ИПР, шкала,
4 упражнения с блокнотами). Составили детальный план (`IMPLEMENTATION_PLAN.md`) и прошли почти
весь его за день.

### Что делали

1. **Phase 0 — аудит эталонов.** Распарсили образец отчёта (`.pptx`) и вытащили курированную
   библиотеку заказчика в `bot/resources/competency_library.json`: литература, онлайн-курсы (с
   URL) и «задания для закрепления» по 6 компетенциям. Разобрали структуру ИПР (методблок +
   таблицы 5 категорий 70/20/10). Зафиксировали шкалу.
2. **Phase 1 — таймкоды.** Whisper переведён в `verbose_json`, сегменты со временем сохраняются

   детерминированно сопоставляются с сегментами → `[мм:сс]` в колонке E блокнота.
3. **Phase 2 — рекомендации = библиотека + LLM.** `competency_content.py` матчит компетенцию из

5. **Phase 4 — ИПР по шаблону.** LLM в том же проходе выдаёт объект `ipr` (цель, действия по

6. **Phase 5–6.** Многоупражненческая агрегация уже была корректна по ТЗ (проверили). Прошли
   чек-лист приёмки.

### Проблемы и как решали

- **PDF не создавался** (LibreOffice возвращал успех, но файла нет) — болезнь первого запуска
  impress с чистым профилем. Добавили явный фильтр, парсинг пути из вывода, glob-fallback и
  повторную попытку.
- **«Таймкоды = 0»** — оказалось, моя проверка смотрела в пустой `sharedStrings`, а openpyxl пишет
  значения инлайн в лист. На деле таймкоды были (31 шт). Ложная тревога.
- **Плохие роли → низкие уровни.** После переезда разметки ролей на gpt-4o-mini роли стали путаться
  (3 голоса, нет диаризации; Whisper склеивает говорящих в одну строку). Подняли модель ролей до
  **gpt-5-mini** + крупнее чанки (6000) + промпт про разрезание склеек → роли стали корректными и
  согласованными, число «+» выросло (54 против почти пустого).

### Результат

- Полный конвейер под ТЗ: таймкоды в блокноте, отчёт-презентация по образцу (PPTX/DOCX/PDF),
  ИПР по таблице-шаблону, курированная библиотека заказчика, отчётные формулировки.
- «Команда моделей»: Whisper (STT) / gpt-5-mini (роли) / gpt-4o-mini (анализ, рекомендации).

### Заметки на будущее

- Прогнать реальный центр из 4 упражнений (Phase 5.3) и сверить отчёт/ИПР с образцом (Phase 6).
- Чанки длинного аудио всё ещё не тестированы на файле >100 минут.
- Whisper склеивает спикеров в одну строку — диаризация (Yandex v3 speaker labeling?) могла бы
  ещё улучшить роли.
- PPTX повторяет структуру образца, но не пиксель-в-пиксель брендированный дизайн.

## 2026-06-06 — Чанки аудио, переезд на gpt-4o-mini, полный прогон сценария

Большой день. Начали с одной фичи, а в итоге перебрали почти весь конвейер и впервые
прогнали полный сценарий упражнения на боевом сервере от и до.

### Что делали

1. **Изучили проект** по `PROJECT_OVERVIEW.md` и коду. Освежили картину: Telegram-бот для
   HR-ассессмента, цепочка аудио → STT → роли → блокнот → отчёт → ИПР.

2. **Нарезка длинного аудио на чанки** (главная задача дня). Сделали `audio_chunking.py`:
   определение длительности через ffprobe (с fallback на ffmpeg), нарезка через ffmpeg с
   перекрытием (overlap), склейка transcript'ов с удалением дублей на стыках. Сначала включили
   для AI Tunnel и NeuroAPI, но владелец уточнил: лимит 25 MB только у AI Tunnel — оставили
   нарезку только там, NeuroAPI вернули на обычное сжатие.

3. **Деплой и инфраструктура.** Настроили `gh auth login`, разобрались с веткой → main → push.
   На сервере: venv в `/opt/asses/.venv`, бот запускается как `python -m bot.main` через systemd.

4. **Полный прогон сценария** одного упражнения: центр → участник → упражнение → аудио →
   роли → блокнот → отчёт → ИПР → PDF.

5. **Переезд LLM на AI Tunnel + gpt-4o-mini** для анализа и разметки ролей (раньше анализ был
   на NeuroAPI deepseek-v4-flash). Дешевле и стабильнее.

6. **Доводка отчёта/ИПР:** LLM-рекомендации вместо шаблона, чистка текстов индикаторов,
   ускорение через параллельные LLM-вызовы, PDF-экспорт через LibreOffice.

### Проблемы и как решали

- **`python -m alembic` не работает** на сервере (`No module named alembic.__main__`).
  → У alembic нет `__main__`. Запускать бинарь из venv: `/opt/asses/.venv/bin/alembic upgrade head`.

- **Бот падал на разметке ролей** сразу после STT: `ValueError: Invalid format specifier`.
  → В `role_labeling.py` JSON-пример `{"segments":...}` стоял внутри f-строки, и Python принял
  фигурные скобки за поле формата. Экранировали `{{ }}`. Давний баг, проявлялся на любой разметке.

- **`/process_exercise` висел ~10 минут и падал молча.**
  → Причина: один LLM-запрос на все 89 индикаторов сразу — NeuroAPI возвращал пустой/обрезанный
  JSON. Плюс handler не ловил ошибку. Сделали: батчи по 20 индикаторов, обработку ошибок в
  handler, диагностику `finish_reason`.

- **Батчи стали проходить, но 0 индикаторов извлекалось.**
  → gpt-4o-mini возвращал валидный JSON, но в разных формах: то голый массив, то обёртка
  `observations`/`indicators`, то словарь по id `{"I001": {...}}`, `evidence` строкой,
  `null` вместо строк. Вместо борьбы с моделью сделали **терпимый парсер** `_coerce_notebook_report`,
  который нормализует любую из этих форм. Ключевой ход — начали **логировать сырой JSON** при
  сбое, и сразу стало видно, что отдаёт модель.

- **Отчёт/ИПР были шаблонные:** «Отработать поведение: <дословный индикатор>. Подготовить 1-2
  ситуации...» — один и тот же хвост у всех пунктов.
  → Сделали `development_advice.py`: gpt-4o-mini генерирует персональные рекомендации и действия
  по модели 70/20/10. Генерация один раз при отчёте, ИПР переиспользует из БД.

- **В «Зонах роста» торчал ассессорский мусор** (скобки-примеры, цитаты, «ставим замеры»,
  «закрываем», «Дополнительный замер:»).
  → Написали regex-чистильщик `clean_indicator_text`. Первая версия оставляла хвосты (обрывалась
  на сокращении «доп.», стряхивала битые скобки/`+`) — доработали: маркеры-вводители примеров
  режем до конца строки, одиночные `)`/`+` убираем.

- **`/process_exercise` ~4 минуты.**
  → Распараллелили батчи блокнота и генерацию рекомендаций через `asyncio.gather` с лимитом
  конкурентности (`ANALYSIS_LLM_MAX_CONCURRENCY`). Сжалось примерно до времени одного батча.

### Результат

- Полный сценарий упражнения работает end-to-end на боевом сервере, качество **боевое**.
- **Роли размечаются правильно** — оцениваемый руководитель = «Участник», ролевой сотрудник и
  ассессор = «Ведущий». Это была историческая боль проекта; gpt-4o-mini справился, даже несмотря
  на то, что имя участника в системе («Иванов Иван») не совпадало с реальным (Маргарита) — модель
  определила оцениваемого по поведению.
- Блокнот заполняется осмысленно, уровни считаются.
- Отчёт и ИПР: реальные компетенции из блокнота, разбивка сильные/зоны роста, **персональные
  LLM-рекомендации**, чистые тексты, PDF на выходе.
- Всё на единой связке **AI Tunnel + gpt-4o-mini** (STT / роли / анализ).

### Заметки на будущее

- **Чанки аудио так и не проверены** на длинном файле (>100 минут) — не было записи. Это
  первый кандидат на следующий раз.
- Имя участника в боевом прогоне нужно вводить настоящее (а не плейсхолдер).
- Уровни компетенций слегка плавают между прогонами — это нормальная вариативность LLM.
- Возможные следующие шаги: агрегация нескольких упражнений на участника, очередь LLM-задач,
  усиление админки, точные шаблоны заказчика для DOCX/PDF.
