# Дневник разработки

Здесь по дням фиксируется живая хронология работы над проектом: что делали, на что
наткнулись, как чинили, что получилось. Не строгая документация (она в `PROJECT_OVERVIEW.md`),
а рассказ — чтобы через месяц можно было вспомнить «а почему мы так сделали».

Формат записи:

```
## YYYY-MM-DD — короткий заголовок дня

### Что делали
### Проблемы и как решали
### Результат
### Заметки на будущее
```

Новые записи добавляются сверху (свежие — первыми).

## 2026-06-14 — Доведение под ТЗ заказчика: таймкоды, библиотека, презентация, ИПР

Получили материалы заказчика (ТЗ + эталоны: образец отчёта на 23 слайда, шаблон ИПР, шкала,
4 упражнения с блокнотами). Составили детальный план (`IMPLEMENTATION_PLAN.md`) и прошли почти
весь его за день.

### Что делали

1. **Phase 0 — аудит эталонов.** Распарсили образец отчёта (`.pptx`) и вытащили курированную
   библиотеку заказчика в `bot/resources/competency_library.json`: литература, онлайн-курсы (с
   URL) и «задания для закрепления» по 6 компетенциям. Разобрали структуру ИПР (методблок +
   таблицы 5 категорий 70/20/10). Зафиксировали шкалу.
2. **Phase 1 — таймкоды.** Whisper переведён в `verbose_json`, сегменты со временем сохраняются
   на записи (миграция `0009`), для чанков время приводится к абсолютному. Цитаты-evidence
   детерминированно сопоставляются с сегментами → `[мм:сс]` в колонке E блокнота.
3. **Phase 2 — рекомендации = библиотека + LLM.** `competency_content.py` матчит компетенцию из
   блокнота к библиотеке; отчёт рендерит Литературу/Курсы/Задания заказчика + LLM-рекомендации.
4. **Phase 3 — отчёт-презентация и мультиформат.** `report_pptx.py` строит презентацию по образцу
   (титул, матрица, сводный слайд, по компетенции — результат/сильные-зоны/рекомендации/задания).
   У бота кнопка выбора формата PPTX/DOCX/PDF. Сильные стороны и зоны роста LLM переписывает в
   «отчётный» стиль.
5. **Phase 4 — ИПР по шаблону.** LLM в том же проходе выдаёт объект `ipr` (цель, действия по
   категориям, ожидаемые результаты); ИПР рендерится как таблица-форма заказчика, ресурсы
   «Самообучения» — из библиотеки.
6. **Phase 5–6.** Многоупражненческая агрегация уже была корректна по ТЗ (проверили). Прошли
   чек-лист приёмки.

### Проблемы и как решали

- **PDF не создавался** (LibreOffice возвращал успех, но файла нет) — болезнь первого запуска
  impress с чистым профилем. Добавили явный фильтр, парсинг пути из вывода, glob-fallback и
  повторную попытку.
- **«Таймкоды = 0»** — оказалось, моя проверка смотрела в пустой `sharedStrings`, а openpyxl пишет
  значения инлайн в лист. На деле таймкоды были (31 шт). Ложная тревога.
- **Плохие роли → низкие уровни.** После переезда разметки ролей на gpt-4o-mini роли стали путаться
  (3 голоса, нет диаризации; Whisper склеивает говорящих в одну строку). Подняли модель ролей до
  **gpt-5-mini** + крупнее чанки (6000) + промпт про разрезание склеек → роли стали корректными и
  согласованными, число «+» выросло (54 против почти пустого).

### Результат

- Полный конвейер под ТЗ: таймкоды в блокноте, отчёт-презентация по образцу (PPTX/DOCX/PDF),
  ИПР по таблице-шаблону, курированная библиотека заказчика, отчётные формулировки.
- «Команда моделей»: Whisper (STT) / gpt-5-mini (роли) / gpt-4o-mini (анализ, рекомендации).

### Заметки на будущее

- Прогнать реальный центр из 4 упражнений (Phase 5.3) и сверить отчёт/ИПР с образцом (Phase 6).
- Чанки длинного аудио всё ещё не тестированы на файле >100 минут.
- Whisper склеивает спикеров в одну строку — диаризация (Yandex v3 speaker labeling?) могла бы
  ещё улучшить роли.
- PPTX повторяет структуру образца, но не пиксель-в-пиксель брендированный дизайн.

## 2026-06-06 — Чанки аудио, переезд на gpt-4o-mini, полный прогон сценария

Большой день. Начали с одной фичи, а в итоге перебрали почти весь конвейер и впервые
прогнали полный сценарий упражнения на боевом сервере от и до.

### Что делали

1. **Изучили проект** по `PROJECT_OVERVIEW.md` и коду. Освежили картину: Telegram-бот для
   HR-ассессмента, цепочка аудио → STT → роли → блокнот → отчёт → ИПР.

2. **Нарезка длинного аудио на чанки** (главная задача дня). Сделали `audio_chunking.py`:
   определение длительности через ffprobe (с fallback на ffmpeg), нарезка через ffmpeg с
   перекрытием (overlap), склейка transcript'ов с удалением дублей на стыках. Сначала включили
   для AI Tunnel и NeuroAPI, но владелец уточнил: лимит 25 MB только у AI Tunnel — оставили
   нарезку только там, NeuroAPI вернули на обычное сжатие.

3. **Деплой и инфраструктура.** Настроили `gh auth login`, разобрались с веткой → main → push.
   На сервере: venv в `/opt/asses/.venv`, бот запускается как `python -m bot.main` через systemd.

4. **Полный прогон сценария** одного упражнения: центр → участник → упражнение → аудио →
   роли → блокнот → отчёт → ИПР → PDF.

5. **Переезд LLM на AI Tunnel + gpt-4o-mini** для анализа и разметки ролей (раньше анализ был
   на NeuroAPI deepseek-v4-flash). Дешевле и стабильнее.

6. **Доводка отчёта/ИПР:** LLM-рекомендации вместо шаблона, чистка текстов индикаторов,
   ускорение через параллельные LLM-вызовы, PDF-экспорт через LibreOffice.

### Проблемы и как решали

- **`python -m alembic` не работает** на сервере (`No module named alembic.__main__`).
  → У alembic нет `__main__`. Запускать бинарь из venv: `/opt/asses/.venv/bin/alembic upgrade head`.

- **Бот падал на разметке ролей** сразу после STT: `ValueError: Invalid format specifier`.
  → В `role_labeling.py` JSON-пример `{"segments":...}` стоял внутри f-строки, и Python принял
  фигурные скобки за поле формата. Экранировали `{{ }}`. Давний баг, проявлялся на любой разметке.

- **`/process_exercise` висел ~10 минут и падал молча.**
  → Причина: один LLM-запрос на все 89 индикаторов сразу — NeuroAPI возвращал пустой/обрезанный
  JSON. Плюс handler не ловил ошибку. Сделали: батчи по 20 индикаторов, обработку ошибок в
  handler, диагностику `finish_reason`.

- **Батчи стали проходить, но 0 индикаторов извлекалось.**
  → gpt-4o-mini возвращал валидный JSON, но в разных формах: то голый массив, то обёртка
  `observations`/`indicators`, то словарь по id `{"I001": {...}}`, `evidence` строкой,
  `null` вместо строк. Вместо борьбы с моделью сделали **терпимый парсер** `_coerce_notebook_report`,
  который нормализует любую из этих форм. Ключевой ход — начали **логировать сырой JSON** при
  сбое, и сразу стало видно, что отдаёт модель.

- **Отчёт/ИПР были шаблонные:** «Отработать поведение: <дословный индикатор>. Подготовить 1-2
  ситуации...» — один и тот же хвост у всех пунктов.
  → Сделали `development_advice.py`: gpt-4o-mini генерирует персональные рекомендации и действия
  по модели 70/20/10. Генерация один раз при отчёте, ИПР переиспользует из БД.

- **В «Зонах роста» торчал ассессорский мусор** (скобки-примеры, цитаты, «ставим замеры»,
  «закрываем», «Дополнительный замер:»).
  → Написали regex-чистильщик `clean_indicator_text`. Первая версия оставляла хвосты (обрывалась
  на сокращении «доп.», стряхивала битые скобки/`+`) — доработали: маркеры-вводители примеров
  режем до конца строки, одиночные `)`/`+` убираем.

- **`/process_exercise` ~4 минуты.**
  → Распараллелили батчи блокнота и генерацию рекомендаций через `asyncio.gather` с лимитом
  конкурентности (`ANALYSIS_LLM_MAX_CONCURRENCY`). Сжалось примерно до времени одного батча.

### Результат

- Полный сценарий упражнения работает end-to-end на боевом сервере, качество **боевое**.
- **Роли размечаются правильно** — оцениваемый руководитель = «Участник», ролевой сотрудник и
  ассессор = «Ведущий». Это была историческая боль проекта; gpt-4o-mini справился, даже несмотря
  на то, что имя участника в системе («Иванов Иван») не совпадало с реальным (Маргарита) — модель
  определила оцениваемого по поведению.
- Блокнот заполняется осмысленно, уровни считаются.
- Отчёт и ИПР: реальные компетенции из блокнота, разбивка сильные/зоны роста, **персональные
  LLM-рекомендации**, чистые тексты, PDF на выходе.
- Всё на единой связке **AI Tunnel + gpt-4o-mini** (STT / роли / анализ).

### Заметки на будущее

- **Чанки аудио так и не проверены** на длинном файле (>100 минут) — не было записи. Это
  первый кандидат на следующий раз.
- Имя участника в боевом прогоне нужно вводить настоящее (а не плейсхолдер).
- Уровни компетенций слегка плавают между прогонами — это нормальная вариативность LLM.
- Возможные следующие шаги: агрегация нескольких упражнений на участника, очередь LLM-задач,
  усиление админки, точные шаблоны заказчика для DOCX/PDF.
